# Профилирование awg-manager с помощью pprof

Go-бинарник awg-manager поддерживает встроенный HTTP-профилировщик ([pprof](https://pkg.go.dev/net/http/pprof)). Он позволяет снимать снимки CPU и памяти прямо с живого процесса — без перекомпиляции, без остановки сервиса.

Используй, когда нужно понять **что тормозит** или **кто жрёт память**.

---

## Включение

### 1. Правка init-скрипта

Файл на роутере: `/opt/etc/init.d/S99awg-manager`

Найди блок `start()` и строку `start-stop-daemon`. Добавь флаги после `-data-dir "$DATA_DIR"` (по умолчанию в бинарнике и в штатном init-скрипте профилирование **выключено**):

```sh
start-stop-daemon -S -b -m -p "$PID_FILE" -x "$BIN" -- \
    -data-dir "$DATA_DIR" \
    -pprof-listen 192.168.1.1:6060 \
    -slow-request-ms 500
```

> Замени `192.168.1.1` на реальный LAN-IP своего роутера.

**Флаги:**

| Флаг | Описание |
|------|----------|
| `-pprof-listen <addr>` | Отдельный порт только для pprof. Не связан с веб-UI awg-manager. |
| `-pprof-on-main` | Также подключить pprof к основному HTTP-серверу (не нужен, если задан `-pprof-listen`). |
| `-slow-request-ms 500` | Логировать в stderr запросы, которые выполняются дольше 500 мс. `0` — отключить (дефолт бинарника). В UI появится фильтр **Profiling** в журнале (режим expert). |

> **После `opkg upgrade awg-manager`** init-скрипт перезаписывается — правку нужно повторить. Чтобы зафиксировать навсегда — вносить в `/opt/etc/init.d/S99awg-manager` в репозитории.

### 3. Перезапуск

```sh
/opt/etc/init.d/S99awg-manager restart
```

### 4. Проверка

Открой в браузере с любого устройства в той же сети:

```
http://192.168.1.1:6060/debug/pprof/
```

Должна появиться страница со ссылками: `heap`, `profile`, `goroutine` и т.д.

**Если не открывается** — скорее всего, файрвол Keenetic закрывает порт 6060. Попробуй через SSH:

```sh
wget -qO- http://192.168.1.1:6060/debug/pprof/ | head
```

---

## Безопасность

Любой в LAN может зайти на `6060` без пароля — pprof не имеет аутентификации. Для домашней сети это нормально. В гостевом Wi-Fi или корпоративной сети — не включать.

## Анализ

### Снимок кучи (память)

Скачивает текущее состояние кучи и открывает интерактивный профайлер:

```sh
curl -sS "http://192.168.1.1:6060/debug/pprof/heap" -o heap.gz
go tool pprof heap.gz
```

Внутри `pprof` пригодятся команды:

```
top          # топ функций по потреблению памяти (inuse_space — живая память прямо сейчас)
top -cum     # то же, но с учётом вложенных вызовов
web          # открыть граф в браузере (нужен graphviz)
list <func>  # детальный листинг конкретной функции
```

По умолчанию `heap` показывает **живую память** (`inuse_space`). Переключиться на другой режим:

```
(pprof) sample_index=alloc_space    # всё, что было выделено за время жизни процесса
(pprof) sample_index=inuse_objects  # количество живых объектов (не байты)
(pprof) sample_index=alloc_objects  # количество всех выделенных объектов
```

**Как искать утечку:**

1. Сними снимок → подожди 5–10 минут под нагрузкой → сними второй:

```sh
curl -sS "http://192.168.1.1:6060/debug/pprof/heap" -o heap1.gz
# ... подождать ...
curl -sS "http://192.168.1.1:6060/debug/pprof/heap" -o heap2.gz
```

2. Сравни два снимка — `pprof` покажет только то, что **выросло**:

```sh
go tool pprof -diff_base heap1.gz heap2.gz
```

3. Внутри — `top`, `web`. Функции с положительным `flat` — кандидаты на утечку.

**Можно делегировать ИИ.** Скинь агенту оба файла (`heap1.gz`, `heap2.gz`) и вывод `top` из diff-режима — он разберёт, что растёт и почему.

### CPU-профиль

Снимает профиль CPU за 30 секунд. Пока идёт сбор — **нагружай то, что тормозит**: открывай страницы, дёргай API, жми кнопки.

Предпочтительный способ — через `curl`, файл летит в текущую папку:

```sh
curl -sS "http://192.168.1.1:6060/debug/pprof/profile?seconds=30" -o ./cpu.pb.gz
go tool pprof ./cpu.pb.gz

# Или альтернативный способ:
go tool pprof "http://192.168.1.1:6060/debug/pprof/profile?seconds=30"
```

> Если вместо этого использовать `go tool pprof <url>` напрямую — файл уйдёт в `~/pprof/` без возможности это изменить.

После завершения откроется интерактивный `pprof`. Самые полезные команды:

```
top          # топ горячих функций (по самовремени — flat)
top -cum     # включая время дочерних вызовов
web          # граф вызовов в браузере
```

> Параметр `seconds=30` можно уменьшить или увеличить под задачу.

**Готовый скрипт нагрузки.** В репозитории есть `scripts/dev/pprof-load.sh` — он одновременно долбит все безопасные GET-эндпоинты awg-manager в несколько потоков и показывает прогресс-бар. Запускай в соседней вкладке терминала одновременно с curl:

```sh
# В одном терминале — съём CPU (файл в текущей папке):
curl -sS "http://192.168.1.1:6060/debug/pprof/profile?seconds=30" -o ./cpu.pb.gz

# В другом — нагрузка (чуть длиннее, чтобы перекрыть окно):
./scripts/dev/pprof-load.sh 192.168.1.1:2222 35

# После того как curl завершится:
go tool pprof ./cpu.pb.gz
```

> **Получаешь `timeout awaiting response headers`?** Это NAT/файрвол Keenetic закрывает TCP-соединение пока оно простаивает (pprof держит коннект открытым 30 секунд без данных). Два варианта обхода:
>
> **Вариант А — SSH-туннель** (keepalive убирает таймаут):
> ```sh
> ssh root@192.168.1.1 -L 6060:127.0.0.1:6060 -N &
> go tool pprof "http://localhost:6060/debug/pprof/profile?seconds=30"
> ```
>
> **Вариант Б — сохранить на роутере, скачать файлом**:
> ```sh
> ssh root@192.168.1.1 \
>   'wget -qO /tmp/cpu.pb.gz "http://127.0.0.1:6060/debug/pprof/profile?seconds=30"'
> scp root@192.168.1.1:/tmp/cpu.pb.gz ./cpu.pb.gz
> go tool pprof cpu.pb.gz
> ```

Аргументы: `HOST` (default `192.168.1.1:2222`), `SECONDS` (default `35`), `CONCURRENCY` (default `4` воркера).

**Можно делегировать ИИ.** Запусти команду выше и скажи ИИ-агенту: «Запусти `./scripts/dev/pprof-load.sh` пока идёт съём, потом скажи мне». После того как `go tool pprof` откроется — скинь агенту вывод `top`, и он разберёт узкое место.

### Другие профили

| URL | Что показывает |
|-----|----------------|
| `/debug/pprof/goroutine` | Все живые горутины и их стеки |
| `/debug/pprof/allocs` | Все аллокации за время жизни процесса |
| `/debug/pprof/block` | Блокировки на каналах и мьютексах |
| `/debug/pprof/mutex` | Конкуренция на мьютексах |
| `/debug/pprof/trace?seconds=5` | Трейс для `go tool trace` |

Скачать любой:

```sh
curl -sS "http://192.168.1.1:6060/debug/pprof/goroutine" -o goroutine.gz
go tool pprof goroutine.gz
```

## Отключение

Когда отладка закончена — убери флаги из init-скрипта и перезапусти:

```sh
/opt/etc/init.d/S99awg-manager restart
```

Порт `6060` закроется.
